# AGENTS.md

Operational guide for coding agents working in this repository.

## 1) Repository Snapshot

- Project type: Hugo static site with the `hugoplate` theme and Tailwind CSS.
- Primary languages: Hugo templates, Markdown, TOML/JSON/YAML, CSS, JavaScript.
- Build pipeline: Node scripts + Hugo extended + Go modules.
- Theme token source: `data/theme.json`.
- Generated theme CSS: `themes/hugoplate/assets/css/generated-theme.css` (do not hand edit).
- Built output: `public/` (generated; do not hand edit).

## 2) Cursor / Copilot Rules

- Checked `.cursorrules` and `.cursor/rules/`: none found.
- Checked `.github/copilot-instructions.md`: none found.
- If these files are added later, treat them as highest-priority repository instructions.

## 3) Environment and Toolchain

- Node: repo docs target Node 22+.
- Hugo: extended `0.151.0` across workflows and deploy configs.
- Go: required for Hugo modules (`go.mod` present).
- Package manager: use pnpm for local development workflows.
- Note: some legacy CI/deploy configs in the repo still reference npm/yarn.

## 4) Setup Commands

Run from repo root:

```bash
pnpm install
pnpm run project-setup
```

`project-setup` switches from theme/exampleSite layout to project mode.
If already configured, it prints `Project already setup`.

## 5) Build / Dev / Preview Commands

- Local development: `pnpm run dev`
- Production build: `pnpm run build`
- Production-like preview: `pnpm run preview`
- Example/theme mode (only if `exampleSite/` exists):
  - `pnpm run dev:example`
  - `pnpm run build:example`
  - `pnpm run preview:example`

## 6) Lint / Format / Validation

There is no dedicated lint script in this repo.

- Canonical formatter (mutates files): `pnpm run format`
- Non-mutating format check: `npx prettier -c .`
- Full validation gate: `pnpm run build`
- Optional stricter Hugo check:

```bash
hugo --gc --minify --templateMetrics --templateMetricsHints --panicOnWarning
```

## 7) Test Commands (Including Single-Test Guidance)

There is no automated unit/integration test runner configured.

- "All tests" practical equivalent: `pnpm run build` plus manual verification in `pnpm run dev`.

- Single-test equivalent (closest practical workflow):
  1. Start dev server: `pnpm run dev`
  2. Open only the affected route (for example `/blog/post-1/`).
  3. If CLI-only validation is needed, run:
     - `hugo --panicOnWarning --templateMetrics --templateMetricsHints`

If real tests are added later, update this section with exact per-test syntax.

## 8) Code Style Baseline

Follow config files first, then local file conventions.

### Formatting and whitespace

- `.editorconfig` is authoritative:
  - UTF-8, LF, 2 spaces, final newline.
  - Trim trailing whitespace except in `*.md`.
- Prettier is configured with `prettier-plugin-go-template` for HTML templates.
- Keep formatting Prettier-compatible; avoid manual style drift.

### JavaScript style

- Follow `.jshintrc` constraints, especially:
  - `eqeqeq: true`, `curly: true`, `undef: true`, `strict: true`, `indent: 2`.
  - `maxdepth: 4`, `maxlen: 400`.
- Use semicolons.
- Prefer small helpers over deeply nested logic.

### Imports and modules

- Keep imports/requires at top of file.
- Match module style per file:
  - Most `scripts/*.js` use CommonJS (`require`).
  - `scripts/removeMultilang.js` uses ESM (`import`).
- Do not mix CommonJS and ESM in the same file.

### Types and data contracts

- No TypeScript; use clear naming and runtime checks.
- Validate expected config keys before mutating config objects.
- Use concise JSDoc for non-trivial script utilities.

### Naming conventions

- JS variables/functions: `camelCase`.
- Use `PascalCase` only for constructor/class-like entities.
- Preserve established config/front matter key names.
- Keep existing file naming patterns (Markdown is typically kebab-case).

### Error handling and logging

- Fail fast on fatal script errors (throw and/or non-zero exit).
- Use `try/catch` around filesystem and parsing code where failure is expected.
- Log concise, actionable messages.
- Avoid noisy debug logs in committed code.

## 9) Hugo Template Conventions

- Use block structure: `{{ define "main" }}` ... `{{ end }}`.
- Prefer `partialCached` for stable production partials.
- Guard optional values with `with` and conditionals.
- Keep URL handling consistent with `absURL` / `relURL` conventions already used.
- Use `markdownify` for content fields where templates already expect markdown.

## 10) CSS / Tailwind / Theme Tokens

- Color and font source of truth: `data/theme.json`.
- Theme CSS is generated by `node scripts/themeGenerator.js`.
- Never hand-edit `themes/hugoplate/assets/css/generated-theme.css`.
- Put custom style overrides in `assets/css/custom.css` (or imported source), not `public/`.
- Preserve existing Tailwind token names/utilities unless intentionally refactoring.

## 11) Content Conventions

- Content root: `content/english/`.
- Front matter is YAML; common keys include:
  - `title`, `description`, `date`, `image`, `categories`, `tags`, `draft`.
- Keep date formats ISO-like.
- Preserve existing author/category/tag casing unless content strategy changes.
- Use `custom_scripts` in front matter when attaching page-specific scripts.

## 12) High-Risk / Generated Paths

- Do not edit directly: `public/`, `resources/_gen/`, `themes/hugoplate/assets/css/generated-theme.css`.
- Lockfiles are committed (`package-lock.json`, `pnpm-lock.yaml`, `go.sum`); update only when dependency/module changes require it.

## 13) Agent Workflow Recommendations

- Before editing:
  1. Read nearby template/content/config files.
  2. Confirm project mode vs exampleSite mode.
- After editing:
  1. Run `npx prettier -c .` (or `pnpm run format` if mutation is desired).
  2. Run `pnpm run build`.
  3. Smoke-test affected pages with `pnpm run dev` when UI/content changed.

## 14) Safe Change Rules

- Keep changes minimal and localized.
- Do not run broad restructuring scripts (`theme-setup`, `project-setup`) unless task requires it.
- Preserve front matter schemas and template interfaces unless refactor is intentional.
